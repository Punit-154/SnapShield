# SMSentry Codebase Review — Bugs, Improvements & Recommendations

Generated by full codebase scan. Organized by severity.

---

## 🔴 Critical Bugs

### 1. Resource Leaks in PrivacyProxyClient (3 locations)
**File:** `app/src/main/java/com/smssentry/deepcheck/proxy/PrivacyProxyClient.kt`
- **Line 46:** `isAvailable()` — `response` is never closed on success path
- **Line 67:** `whois()` — `response` is never closed
- **Line 85:** `fetchPage()` — `response` is never closed

OkHttp responses **must** be closed. Leaked connections exhaust the connection pool and eventually cause `IOException: Connection pool shut down`.

**Fix:** Wrap every `call.execute()` in `response.use { ... }`.

---

### 2. Resource Leak in FetchPageTool
**File:** `app/src/main/java/com/smssentry/deepcheck/tools/FetchPageTool.kt`
- **Lines 22-24:** If `response.body?.string()` throws, `response.close()` is never called.

```kotlin
// Current (leaky):
val response = client.newCall(request).execute()
val body = response.body?.string()?.take(500) ?: ""
response.close()

// Fix:
client.newCall(request).execute().use { response ->
    val body = response.body?.string()?.take(500) ?: ""
    return@withContext "Page content (first 500 chars): $body"
}
```

---

### 3. Resource Leak in OfficialSitesRepository
**File:** `app/src/main/java/com/smssentry/deepcheck/data/OfficialSitesRepository.kt`
- **Lines 14-17:** `inputStream` is opened but never closed if `reader.readText()` or the JSON parse throws.

```kotlin
// Fix:
context.assets.open("official_sites.json").use { inputStream ->
    BufferedReader(InputStreamReader(inputStream)).use { reader ->
        val raw = reader.readText()
        sites = try { json.decodeFromString(raw) ... } catch ... 
    }
}
```

### 4. JSON Injection via String Interpolation
**File:** `app/src/main/java/com/smssentry/deepcheck/session/DeepCheckSession.kt`
- **Line 301:** `"""{"urls":[$urlsJson]}"""` — URLs containing `"` break the JSON.
- **Line 313:** `"""{"domain":"$domain"}"""` — domain containing `"` breaks the JSON.
- **Line 324:** `"""{"sms_text":"$smsText","urls":[$urlsJson]}"""` — `smsText` containing `"` breaks the JSON.
- **Line 337:** `"""{"claimed_entity":"$claimedEntity","linked_domain":"$domain"}"""`

**Fix:** Use `Json.encodeToString()` or manual escaping for all JSON construction.

---

### 5. Notification ID Overflow
**File:** `app/src/main/java/com/smssentry/sms/SmsReceiver.kt`
- **Line 102:** `System.currentTimeMillis().toInt()` — Long-to-Int truncation can produce negative IDs or collisions. Multiple rapid notifications can overwrite each other.

**Fix:** Use `AtomicInteger` counter or `notificationManager.notify(id, ...)` with a proper ID scheme.

---

### 6. Thread-Safety: ModelRepository.getEngine()
**File:** `app/src/main/java/com/smssentry/deepcheck/data/ModelRepository.kt`
- **Lines 90-95:** `getEngine()` is not synchronized. Two concurrent callers can both see `cachedEngine == null` and create two engine instances. Only one will be retained; the other leaks native memory (LiteRT engines hold native resources).

**Fix:** Add `@Synchronized` or use `Mutex`.

---

### 7. LlmInferenceEngine Provider Not Scoped
**File:** `app/src/main/java/com/smssentry/di/AppModule.kt`
- **Lines 96-102:** `provideLlmInferenceEngine()` lacks `@Singleton`. Every injection creates a new engine instance. Since `LlmInferenceEngine?` is injected into `DetailViewModel` via `ModelRepository`, this might be masked, but the provider itself creates a new engine every time it's called.

**Fix:** Add `@Singleton` or remove this provider entirely (DetailViewModel gets the engine via `modelRepository.getEngine()`).

## 🟠 Bugs / Functional Issues

### 8. Triple Duplication of Model Management
Three classes do the same thing:
- `ModelManager.kt` (lines 24-27: `isModelDownloaded()`, lines 54-64: `getLlmEngine()`)
- `ModelRepository.kt` (lines 55-57: `isModelDownloaded()`, lines 90-95: `getEngine()`)
- `ModelDownloadManager.kt` (lines 67-69: `isModelDownloaded()`, lines 71-76: `isOnWiFi()`)

Constants are also duplicated (`MODEL_URL`, `MODEL_FILE_NAME`, `MIN_FILE_SIZE_BYTES`, `MODEL_SHA256` exist in both `DeepCheckConfig.kt` and `ModelDownloadManager.kt`).

**Recommendation:** Consolidate into `ModelRepository` only. Remove `ModelManager` and `ModelDownloadManager`.

---

### 9. Duplicate `classifyByRules()` Implementations
- `InboxViewModel.kt:175-185` — indicator list: `["urgent", "verify", "suspended", "click here", "congratulations", "won", "prize", "lottery"]`
- `RealSMSSentryAI.kt:60-88` — indicator list: `["urgent", "verify", "suspended", "click here", "congratulations"]`

Different lists mean different classification results depending on code path.

**Recommendation:** Extract single shared classifier.

---

### 10. DetailScreen Perpetual Loading on Deleted Message
**File:** `app/src/main/java/com/smssentry/ui/detail/DetailViewModel.kt`
- **Lines 57-64:** If `smsRepository.getMessageById(smsId)` returns null (message was deleted between navigation), `_message` stays null forever. The screen shows a loading spinner with no way out.

**Fix:** Emit an error state or navigate back.

---

### 11. No Notification Tap Action
**File:** `app/src/main/java/com/smssentry/sms/SmsReceiver.kt`
- **Lines 94-101:** Notifications have no `PendingIntent`. Users see "Scam Detected!" but can't tap to see details or take action.

**Fix:** Add `setContentIntent()` with a `PendingIntent` to open `DetailScreen` for the relevant message.

### 12. Swipe-to-Dismiss Doesn't Actually Delete SMS
**File:** `app/src/main/java/com/smssentry/ui/inbox/InboxViewModel.kt`
- **Lines 134-138:** `deleteMessage()` only removes from the local `_allMessages` list. The SMS remains in the device inbox. The "Undo" feature only restores the local list item.

This is misleading to users who expect message deletion.

## 🟡 Performance Issues

### 13. No Debounce on SMS Content Observer
**File:** `app/src/main/java/com/smssentry/ui/inbox/InboxViewModel.kt`
- **Lines 98-106:** `SmsContentObserver` fires `refreshMessages()` on every SMS change. Bulk operations (e.g., importing SMS) will trigger dozens of rapid re-queries.

**Fix:** Add debounce (e.g., `debounce(500)` on the observer callback) or use `conflate` in the coroutine.

---

### 14. InboxViewModel classifyByRules on Main Thread
**File:** `app/src/main/java/com/smssentry/ui/inbox/InboxViewModel.kt`
- **Lines 158-165:** `realMessages.map { classifyByRules(it.text) }` runs synchronously on the main thread inside the coroutine. For large inboxes (50+ messages), this adds latency to UI rendering.

**Fix:** Move the map operation to `Dispatchers.Default` or use `parallelMap`.

---

### 15. OfficialSitesRepository Loads JSON on Construction
**File:** `app/src/main/java/com/smssentry/deepcheck/data/OfficialSitesRepository.kt`
- **Lines 12-33:** The entire JSON file is read and parsed in the constructor, which runs on the main thread during Hilt injection at app startup.

**Fix:** Make it a `suspend fun` or lazy-load it.

---

### 16. SMSSentryApp Creates Duplicate CoroutineScope
**File:** `app/src/main/java/com/smssentry/SMSSentryApp.kt`
- **Line 17:** `private val appScope = CoroutineScope(Dispatchers.IO + SupervisorJob())` — but `AppModule.kt:107-108` already provides `@ApplicationScope CoroutineScope`. Two independent scopes with no shared lifecycle management.

**Fix:** Use the DI-provided scope, or inject it into the Application.

## 🟡 Security Concerns

### 17. Unencoded URL Parameters
**File:** `app/src/main/java/com/smssentry/deepcheck/proxy/PrivacyProxyClient.kt`
- **Line 64:** `"$baseUrl/whois?domain=$domain"` — domain not URL-encoded
- **Line 82:** `"$baseUrl/fetch-page?url=$url"` — URL not URL-encoded

Special characters (`&`, `=`, `?`, spaces) in input break the request or enable parameter injection.

**Fix:** Use `HttpUrl.Builder` or `URLEncoder.encode()`.

---

### 18. FetchPageTool SSRF Risk
**File:** `app/src/main/java/com/smssentry/deepcheck/tools/FetchPageTool.kt`
- **Lines 17-29:** Any `http://` or `https://` URL is fetched without restriction. If the LLM is tricked into calling `fetch_page` with `http://169.254.169.254/latest/meta-data/` (cloud metadata endpoint), it could leak instance credentials.

**Fix:** Block private IP ranges, localhost, and link-local addresses.

### 24. InboxViewModel Missing Null Safety for Classification
**File:** `app/src/main/java/com/smssentry/ui/inbox/InboxViewModel.kt`

### 25. HistoryDao Missing Index
**File:** `app/src/main/java/com/smssentry/deepcheck/data/HistoryEntry.kt`
- No `@Index` annotation on `timestamp` column. The `pruneOlderThan()` query does a full table scan.

**Fix:** Add `@Entity(tableName = "history", indices = [Index("timestamp")])`.

### 26. AllowlistDao Missing Index
**File:** `app/src/main/java/com/smssentry/deepcheck/data/AllowlistEntry.kt`
- No composite index on `(id, type)`. The `containsSender()` and `containsDomain()` queries scan the table.

**Fix:** Add `@Entity(tableName = "allowlist", indices = [Index(value = ["id", "type"], unique = true)])`.

### 27. DeepCheckConfig Constants vs ModelDownloadManager Constants

- `ModelDownloadManager.kt` defines the same constants in its `companion object`

These can diverge. One should reference the other.
